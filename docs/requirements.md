
# 小諸そば メニューガチャ PWA 検討資料

## 1. 背景・モチベーション

- 平日 3 回小諸そばを利用しており、半年以上継続している。
- 「たまには別のものを食べたい」「いつも同じメニューを選んでしまう」という単調さを破りたい。
- PWA(スマホで全画面アプリ風に動くもの)を試したい技術的興味も兼ねる。
- 個人用ツール。マルチユーザー想定なし。

## 2. ドメイン前提

### 2.1 店舗利用に関する観察

- 主食の単位は「麺メニュー」「丼単品」「セット(丼+麺)」の3系統。
- 「ライス」を単品注文している人は店頭でほぼ見ない(=ニーズが薄い)。
- 「いなり」は副菜として人気が高く、「そば+いなり」のようにメイン麺と一緒に頼む人がいる。
- 「満腹セット」(大きい丼+麺)を頼む人に追加で大盛りを乗せるのは不自然(満腹セットを選んだ時点で大盛り相当)。
- 一方、「お蕎麦と丼のセット」(ミニ丼+麺)や麺単品に大盛りを足すのは普通の組み合わせ。

### 2.2 メニュー構造に関する観察(PDF より)

- 同じ品名でも **麺種別ごとにカロリーが別の行として定義** されている(例: かけそば 330kcal / かけうどん 282kcal)。
- 麺種別は以下の 4 種類が登場する。
  - そば(温)
  - うどん(温)
  - せいろ(=冷たいそば)
  - うどんせいろ(=冷たいうどん)
- すべての品名が 4 種類のバリアントを持つわけではない。
  - 「もり」「ざる」「冷したぬき」などは品名自体に温冷を含むため、そば/うどんの 2 バリアントのみ。
  - 「かけ」「たぬき」「きつね」など温メニューも、せいろバリアントは持たない品がある。
  - 「かき揚げ」「いか天」「えび天」「鳥から」「山かけ」など揚げ物・トッピング系は 4 バリアントを持つ。
- 「丼単品(ごはんもの)」は麺を持たないので麺種別は無関係(独立)。
- セットは 2 階層に分かれる。
  - **満腹セット**: 大きい丼+麺(例: 上天丼セット 1028kcal/そば)
  - **お蕎麦と丼のセット**: ミニ丼+麺(例: かき揚げ丼セット 871kcal/そば)
  - 両方とも 4 麺バリアント(そば/うどん/せいろ/うどんせいろ)を持つ。
- 「大盛り」は加算カロリーオプションとして PDF に記載されており、麺種別ごとに値が異なる。
  - かけそば +141 / かけうどん +87 / せいろ +140 / うどんせいろ +95
- 「麺のみ:そば」「麺のみ:うどん」もカロリー表に存在するが、店頭注文の主役にはなり得ない。

## 4. 機能要件

### 4.1 体験(UX)

- ワンスクリーン即ガチャ型。設定軸ボタンとガチャを引くボタンが常時見える。
- 起動時は常にデフォルト(ふつう+そば)からスタート。前回条件は復元しない(=毎日同じタップ数で「今日の気分」をリセットできる)。
- 結果画面の構成:
  - メニュー名(主役・大きく表示)
  - 画像枠(中身は保留、いらすとや or AI 生成は別途検討)
  - 「もう一回」ボタン(同条件でリロール)
- 結果画面で価格・カロリーを出すかは未決(本人判断保留)。
- ガチャ演出の具体は別 AI で検討予定。

### 4.2 選択軸

| 軸   | UI 表示   | 補足    |
| --- |  | -------- |
| 量   | かるめ / ふつう / おおめ / がっつり | UI ラベルのみ。内部カロリー閾値は持つが画面には出さない。 |
| 麺種別 | そば / うどん / 丼単品    | 排他選択 3 ボタン。|
| 温冷  | (UI なし) | データに含めるが選ばせない。抽選任せ。  |
| 大盛り | (UI なし) | 内部ロジックで確率付与。    |

### 4.3 抽選ロジック

- ベース: 完全ランダム(重み付けなし、重複除外なし)。
- 麺軸「そば」「うどん」を選んだ場合、抽選プールには **麺メニュー + 満腹セット + ミニ丼セット** を含める(同麺種別のもの)。
- 麺軸「丼単品」を選んだ場合、抽選プールは **ごはんもの(丼単品)のみ**。
- 量軸 × 麺軸の組み合わせで成立しないものは UI 上で選択不可にする(例: かるめ × 丼単品 は丼のカロリーが高いため選べない想定)。
  - 実装手段の案として「カロリー閾値変更に追従させるため、対象 0 件のボタンを動的に無効化する」が考えられるが、確定はしていない(検討事項)。
- **大盛り付与**:
  - 対象カテゴリ: 麺メニュー / ミニ丼セット のみ。
  - 確率: 量軸「おおめ」「がっつり」のときのみ発火対象。具体確率は実装時に定数で決める。
  - 加算カロリー: 麺種別ごとの大盛りカロリーを加算。
- **いなり付与**:
  - 対象: 麺メニュー / ミニ丼セット のみ(満腹セット・丼単品には付与しない)。
  - ガード条件: ベースメニューカロリー + 160kcal が、選択された量カテゴリの上限を超えない場合のみ発火対象とする(=「ふつう」を選んだのに合計が「おおめ」相当になる事故を防ぐ)。
  - 確率: 25%(定数として 1 ヶ所で管理。後から調整可能)。

### 4.4 抽選対象に含めないもの

| 項目| 理由|
| - | -- |
| 麺のみ   | 「もり」「ざる」と実質同義で、抽選プールに含める意味が薄い。 |
| ライス   | 店頭で単品注文される実態がほぼない。    |
| いなり(単独)  | サイドとしての付与専用で、メイン結果としては出さない。  |

## 5. データ定義(論理名ベース)

リポジトリに同梱する静的 JSON 上で表現する論理項目の整理(DB ではない)。

### 5.1 メニュー一覧

1 件 = 1 注文単位。バリアント(温冷/麺種別)は別エントリとして展開する。

| 論理項目名| 取り得る値   | 補足|
| -- | - | -- |
| メニュー識別子 | 文字列(重複なし)  | 例: 「かき揚げそば」「かき揚げせいろ」を別 ID として持つ。 |
| 表示名    | 文字列  | UI に出すラベル。「かき揚げそば」「上天丼セット(そば)」等。 |
| カテゴリ  | 麺 / 丼単品 / 満腹セット / ミニ丼セット    | 抽選プール分岐と大盛り・いなり付与可否の判定に使用。   |
| 麺種別    | そば / うどん / なし   | 「丼単品」カテゴリのときは「なし」。    |
| 温冷 | 温 / 冷 | 「丼単品」カテゴリでの扱いは保留(値を持たない/固定値のどちらか)。|
| カロリー  | 整数(kcal)   | PDF 記載値をそのまま転記。    |
| 大盛り加算カロリー | 整数(kcal) または なし  | 麺種別と「温/冷」の組み合わせに対応する大盛り加算値。   |

件数の見積もり: 約 100 件。PDF と 1:1 で対応するため手動転記でメンテ可能。

### 5.2 サイドメニュー(いなり)

いなりは 1 件のみ独立で保持。

| 論理項目名 | 値    |
| - | -- |
| 表示名| いなり(二個) |
| カロリー   | 160kcal |

### 5.3 量カテゴリ閾値(定数)

閾値は定数として 1 ヶ所に集約(別ファイル or 同 JSON 内、実装時に決める)。

| 量カテゴリ | カロリー下限 | カロリー上限 |
| - | - | - |
| かるめ| (未定)  | (未定)  |
| ふつう| (未定)  | (未定)  |
| おおめ| (未定)  | (未定)  |
| がっつり   | (未定)  | なし(上限なし) |

※ 具体値は実装後、メニュー分布を見ながら調整する。

### 5.4 確率設定(定数)

| 項目   | 値    |
| - | -- |
| いなり付与確率   | 25%   |
| 大盛り付与確率   | 未定(実装時に決定) |

## 6. インフラ・技術スタック

| 領域| 採用 | 経緯・理由|
| - | -- | -- |
| ホスティング  | Cloudflare Workers + Static Assets(`wrangler.jsonc`)   | Cloudflare 公式が今後一本化していく方針なので Pages ではなく Workers 側。  |
| フロント FW   | Astro| 個人的に使いたいから(本人決定)。静的書き出しのみで十分。   |
| 言語| TypeScript| データに型を当てたいため。|
| バックエンド  | なし | 抽選はクライアント側で完結。データも静的 JSON 同梱。   |
| データ保持    | リポジトリに静的 JSON 同梱    | PDF は固定的なので動的取得は過剰。改定はコミットで再デプロイで終わる。|
| PWA| 最低限セット(manifest + 192/512 アイコン + standalone + テーマカラー + Service Worker オフラインキャッシュ) | 「ホーム画面追加で全画面起動」を満たす最小限。iOS 専用 splash 等は今回見送り。 |
| 状態管理 | なし(起動時毎回リセット) | 毎日同じ条件をデフォルトにしたいので localStorage 復元はしない。  |

## 7. 抽選フロー(擬似コード相当の流れ)

1. ユーザーが量軸 + 麺軸を選択。
2. UI 側で、各ボタンに対し「対象 0 件にならないか」を都度判定して無効化表示。
3. 「引く」ボタン押下。
4. 抽選プールを構築:
   - 麺軸「そば/うどん」の場合: 該当麺種別の **麺メニュー + 満腹セット + ミニ丼セット**。
   - 麺軸「丼単品」の場合: **丼単品** のみ。
5. プールから量軸のカロリーレンジでフィルタ。
6. 残ったメニューから一様乱数で 1 件選択。
7. **大盛り判定**:
   - カテゴリが 麺 or ミニ丼セット かつ 量軸が おおめ/がっつり ならば、大盛り確率で発火。
   - 発火時はカロリーに加算カロリーを足す。
8. **いなり判定**:
   - カテゴリが 麺 or ミニ丼セット のとき。
   - ベースカロリー + 160 が量カテゴリ上限を超えなければ、25% で付与。
9. 結果画面に選ばれたメニューを表示(大盛り/いなりが付与された場合の表記方法は保留)。

## 8. 設計上の判断と経緯(意思決定ログ)

後から判断を変えるときに「なぜそうしたか」が分かるよう、主要な選択をログ化しておく。

| トピック  | 採用  | 却下した選択肢  | 経緯    |
| ---------- | - | - | ---------- |
| UX 構造 | ワンスクリーン即ガチャ   | ステップウィザード / 設定保存型  | 平日 3 回利用のヘビーユーザーのため摩擦最小化最優先。条件変更の柔軟性も維持したい。   |
| 起動時の前回条件復元 | しない(常にデフォルト)  | localStorage 復元 / URL クエリ復元  | 「今日の気分でリセットしたい」が優位。毎回デフォルトから始めるほうが「気分転換ツール」としての性格に合う。   |
| 麺軸 UI | 3 ボタン排他  | 4 ボタン(セット独立) / 麺軸 + 「丼を含める」トグル    | UI を単純に保つ。セットは「そば/うどん」選択時の抽選プールに含めることで実質カバー。  |
| 温冷の UI 露出  | 出さない| 3 択ボタン / 季節自動判定    | ガチャの楽しみは温冷のランダム要素にもあると判断。タップ数も増えない。 |
| 量の閾値  | 内部のみ、実装後決定    | UI に kcal 表示 / 初期から固定   | 文字数多くなる(本人指摘)。閾値は定数 1 ヶ所に集約しておけば後から書き換えで済む。   |
| 抽選ロジック| 完全ランダム   | 直近選択を除外 / 人気重み付け   | データなし。シンプルで「ガチャらしさ」が出る。同じものが出たら「もう一回」で済む。|
| メニューデータ粒度  | 1 メニュー 1 エントリ | 品名 + バリアント入れ子 / トッピング + ベース分解| 抽選コードが filter→random で最短。PDF と 1:1 突き合わせ可能。データ重複は静的 JSON なので問題なし。 |
| セット階層 | カテゴリで分離(満腹セット/ミニ丼セット)   | 単一 set カテゴリ + size フラグ / 区別しない    | 「満腹セットには大盛り付けない」というドメイン制約を表現するため。   |
| 大盛り対象 | 麺単品 + ミニ丼セットのみ| 麺単品のみ / 全カテゴリ | 「ミニ丼セット + 麺大盛り」は現実にあり得る組み合わせのため。満腹セットは丼が既に大きいので不要。 |
| いなり付与 | 量カテゴリ上限超過しない場合のみ 25%    | 一律 30% / 量軸別確率| カロリー閾値で量を定義しているのに、確率付与で上限を超えると整合性が壊れる(本人指摘)。ガード条件で一貫性を担保。    |
| データ形式 | 静的 JSON 同梱    | Workers KV / クライアントから PDF を fetch | 個人アプリで動的更新ニーズなし。PDF はレイアウト固定ではなく将来変わると壊れるリスクがある。手動転記の方が安定。   |
| ホスティング| Workers + Static Assets | Pages / Workers にインライン HTML  | Cloudflare の今後の推奨パスが Workers + Static Assets に寄っているため。  |

## 9. 保留事項(別途検討)

- **量カテゴリのカロリー閾値の具体値**: 実装後、メニュー分布を見て決定。各量カテゴリで該当メニューが偏らないことを確認しつつ調整。
- **ガチャ演出**: 演出時間・アニメーション内容は別 AI で検討予定。毎日使うため過剰演出は避けたいが、「引いた感」は欲しい。
- **画像素材**: いらすとや / AI 生成 / その他、未定。素材方針が決まるまで結果画面の画像枠はプレースホルダーで開発を進める。
- **大盛り付与確率の具体値**: 実装時に定数で設定。
- **言語(TypeScript / JavaScript)**: 未決。
- **価格・カロリーを結果画面に出すか**: 未決。
- **「丼単品」カテゴリの温冷値の持ち方**: 値なし / 「温」固定 のどちらにするか未決。
- **大盛り・いなり付与時の結果画面表記方法**: 未決。
- **量軸 × 麺軸の不成立組み合わせを UI で無効化する方法**: ハードコード or 動的判定 のどちらにするか未決。
